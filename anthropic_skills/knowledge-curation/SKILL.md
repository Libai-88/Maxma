# 知识库整理工作流

把用户散落的文档（Markdown / 代码 / PDF / 网页摘录）整理进 Maxma 知识库，**确保每条都能被 `kb_search` 检索到**。

## 适用场景

- 用户说"我有很多笔记散在各处，想统一管理"
- 用户已经往 KB 里塞过文档，但搜不到
- 用户想给已有文档重新分类 / 打标签

## 工作流程

### 1. 盘点现有资产

- 用 `kb_list_documents` 列出当前 KB 所有文档，看哪些已经入库、哪些有 `index_error`。
- 用 `ask_user_qa` 询问用户：还有哪些散落的笔记文件？（让用户给目录或文件列表）
- 用 `file_search` 在用户给出的目录里递归找 `.md / .txt / .pdf / .docx`，列清单给用户确认。

### 2. 分类与去重

- 按主题给每份文档打 `section`（标签），例如：`论文 / 工作笔记 / 代码片段 / 学习资料`。
- 若发现 KB 里已有同名 / 内容相似的文档（用 `kb_search` 按标题前几句搜），用 `ask_user_qa` 问用户：覆盖 / 跳过 / 合并？
- 同主题的短笔记可以合并成一份长文档（减少 KB 噪声），合并时保留原文标题作为小节标题。

### 3. 入库并立即自检

**每份文档入库后必须立即自检**，不要批量入库后才查：

```
循环：
  1. kb_add_document(title, content, section, source_path)
  2. 检查返回值：
     - status == "indexed"        → ✅ 索引成功
     - status == "metadata_only"  → ⚠️ 索引失败，看 warning 字段
     - status == "error"          → ❌ 入库失败
  3. 若索引失败：
     a. 把 warning 原文回显给用户
     b. 用 file_search 找日志 %APPDATA%/MaxmaHere/logs/，读 [rag] / [kb] 相关行
     c. 告诉用户具体原因（ONNX 模型缺失 / chromadb 损坏 / 文本太短等）
     d. 不要继续塞文档——先修好索引再继续
  4. 若索引成功，用 kb_search 按文档标题搜一次，确认能搜到。
```

> 关键：**`metadata_only` 不是成功**。这条文档存了但搜不到，等于没存。必须看到 `indexed` 才算成功。

### 4. 标签与命名规范

- 文档标题：`[主题] 具体标题.md`，如 `[论文] GPT-4 技术报告笔记.md`
- `section` 字段用小写英文或简短中文，保持一致（不要一会 `论文` 一会 `paper`）
- `source_path` 填原始文件绝对路径，便于回溯

### 5. 检索验收

整理完成后，让用户给出 3–5 个"应该能搜到"的查询词，逐一 `kb_search` 验证：

- 搜得到 → 报告相似度分数
- 搜不到 → 分析原因（文本太短 / 关键词没出现 / 索引未建立）

最后给出汇总：

```
本次整理：
  - 新增文档：N 份（其中 M 份索引成功）
  - 跳过/合并：K 份
  - 索引失败：L 份（原因列表）
  - 检索验收：X/5 通过
```

## 注意事项

- **不要静默吞 `metadata_only`**：这是 KB 整理最常见的坑——文档进去了但搜不到。每条都要看 `status` 字段。
- **小批量 + 即时自检**：不要一次性塞 50 份文档，那样索引失败时无法定位是哪份触发的。
- **保留原文备份**：入库前确认原文件还在磁盘上，KB 里的内容只是副本。
- **诚实汇报失败**：如果索引引擎整体不可用（ONNX 未装 / chromadb 损坏），明说"当前 KB 索引不可用，文档已保存元数据但无法语义检索"，不要假装成功。
- **不要修改用户原文**：整理时只在 KB 里建副本，磁盘上的原文件不要动。

## 推荐工具组合

| 阶段 | 主用工具 |
|------|---------|
| 盘点 | `kb_list_documents` / `file_search` |
| 澄清 | `ask_user_qa` |
| 入库 | `kb_add_document` |
| 自检 | `kb_search` |
| 排查 | `file_search`（找日志）/ `file_read` |
| 验收 | `kb_search` |
