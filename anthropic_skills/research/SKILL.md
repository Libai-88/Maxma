# 网络研究工作流

围绕用户的问题，做一次系统性的网络调研，输出一份有来源、有交叉验证、有结论的研究简报。

## 适用场景

- 用户说"帮我调研一下 X 现状"
- 用户要做技术选型 / 竞品分析 / 市场调研
- 用户想深入了解某个主题但不知道从何查起

## 工作流程

### 1. 明确研究问题

用 `ask_user_qa` 问清（一次问完）：

1. **核心问题**：一句话描述你想搞清楚什么？
2. **深度**：概览（5 分钟读完）/ 标准（15 分钟）/ 深度（30 分钟+）
3. **用途**：自己了解 / 汇报给老板 / 做决策依据
4. **已知信息**：你已经知道哪些？（避免重复研究）

> 不要跳过这步。"调研一下 AI" 和 "调研一下 2026 年国产 LLM 在金融场景的落地情况" 是完全不同的任务。

### 2. 拆解子问题

把核心问题拆成 3–5 个子问题。例如：

核心问题：*Should we adopt Rust for our new microservice?*

子问题：
1. Rust 在微服务场景的性能优势有多大？
2. Rust 的开发效率 vs Go / Python 如何？
3. Rust 微服务的生态（web 框架 / ORM / observability）成熟度？
4. 团队学习曲线和招聘难度？
5. 已知的大厂案例和踩坑经验？

把子问题列给用户确认后再继续。

### 3. 系统搜索

对每个子问题用 `tavily_search` 搜索：

```
搜索策略：
  - 每个子问题 3–5 个不同关键词组合
  - 中英文各搜一轮（中文用百度/知乎视角，英文用 Google/HN 视角）
  - 优先：官方文档 > 权威媒体 > 行业报告 > 博客 > 论坛
  - 时效性：技术类优先近 1 年，理论类可放宽
```

对每个搜索结果：
- 用 `tavily_extract` 抓取正文（不要只看摘要）
- 快速判断相关性：高度相关 / 部分相关 / 无关
- 无关的直接跳过，不要浪费时间

### 4. 交叉验证

**单一来源不可信**。每个关键结论必须至少 2 个独立来源：

```
结论 A：Rust 比 Go 快 2–3 倍
  来源 1：[benchmark 文章](url1) — 2025-12，TechEmpower
  来源 2：[公司博客](url2) — 2025-10，某公司实测
  来源 3：[Reddit 讨论](url3) — 多人评论一致
  ✅ 三源一致，可信
```

如果来源冲突，明确标注分歧：

```
结论 B：Rust 开发效率如何？
  来源 1：[博客](url1) 说"和 Go 差不多"
  来源 2：[调查](url2) 说"比 Go 慢 30%"
  ⚠️ 来源冲突，可能因项目复杂度而异
```

### 5. 综合成报告

用 `file_write` 生成 Markdown 报告：

```markdown
# 研究简报：{核心问题}

## TL;DR
[3–5 句话结论]

## 研究范围
- 子问题列表
- 信息来源数量 / 类型
- 研究日期

## 主要发现

### 发现 1：[一句话结论]
[详细论证，引用来源]
来源：[1][2][3]

### 发现 2：……

## 分歧与不确定
[来源冲突的地方，以及可能原因]

## 建议
[基于发现的可执行建议，标注确定性]

## 来源清单
[1] 标题 - URL - 日期
[2] ……
```

### 6. 入库（可选）

询问用户是否把研究简报存入知识库（`kb_add_document`，section 填 `研究简报`）。

## 注意事项

- **不要只看搜索摘要**：摘要常断章取义，必须 `tavily_extract` 抓全文。
- **交叉验证是底线**：任何结论只有 1 个来源的，标注"单源未验证"。
- **时效性**：技术类信息超过 2 年基本失效，标注日期。
- **利益相关**：厂商博客夸自己产品的结论要打折，优先看第三方。
- **不要堆砌来源**：5 个高质量来源 > 20 个低质量来源。
- **诚实说"没找到"**：如果某个子问题搜不到可靠信息，明说，不要硬凑。
- **尊重付费墙**：遇到付费文章，不要试图绕过，换免费来源。

## 推荐工具组合

| 阶段 | 主用工具 |
|------|---------|
| 澄清问题 | `ask_user_qa` |
| 搜索 | `tavily_search` |
| 提取正文 | `tavily_extract` |
| 浏览页面 | `browse`（playwright） |
| 写报告 | `file_write` |
| 入库 | `kb_add_document` |
