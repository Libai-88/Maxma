# 论文复现工作流

把一份论文笔记（Markdown / PDF 摘录 / 代码片段）转化成可运行、可验证的复现项目。

## 适用场景

- 用户提供一份 `.md` / `.pdf` 笔记，要求"按这个复现一下"
- 用户已上传论文到知识库，希望基于笔记内容做工程实现
- 用户想验证某篇论文的核心 claim 是否站得住脚

## 工作流程

### 1. 读取并解析笔记

- 用 `file_read` / `file_search` 读取用户指明的笔记文件（绝对路径）。
- 若用户说"已经在知识库了"，先用 `kb_search` 按论文标题/关键词检索，拿到 `document_id` 后再用 `kb_search` 取片段。
- 提取四类关键信息（在回复里显式列出，让用户确认）：
  1. **核心 claim**：论文主张什么？（一句话）
  2. **方法骨架**：算法 / 架构 / 数据流的步骤序列
  3. **关键超参与数据**：学习率、batch、数据集、模型尺寸
  4. **评估指标**：用什么指标、目标值是多少

> 如果四类信息有任何缺失，**先停下来用 `ask_user_qa` 询问**，不要凭猜测往下走。

### 2. 规划实现

- 把方法骨架拆成可独立验证的模块（建议 3–6 个），例如：`data_loader` / `model` / `train_loop` / `eval` / `utils`。
- 用 `todo_add` 把每个模块登记为待办，便于用户追踪进度。
- 与用户确认技术栈（PyTorch / NumPy / 从零实现 …），默认 PyTorch + Python 3.13。

### 3. 逐模块实现 + 单元验证

每个模块按"写 → 跑 → 修"循环：

1. 用 `file_write` 创建模块文件（路径放在用户指定的工作目录，默认 `D:\Users\<user>\Documents\repro\<paper-slug>\`）。
2. 用 `run_python` 写一个最小 smoke test（构造 1 条假数据，跑通 forward / 一轮训练），确认无报错。
3. 若报错，用 `file_edit` 精确修复（不要重写整个文件）。
4. 模块通过后，用 `todo_complete` 标记完成。

> 重要：每个模块都要有独立 smoke test，**不要把所有模块写完才一起跑**——那样定位问题成本极高。

### 4. 整体联调

- 写 `main.py` 把模块串起来，用小数据集（CPU 可跑、1–2 epoch）跑一次完整流程。
- 用 `run_python` 执行，捕获 stdout / stderr。
- 把实际指标和论文声称的指标列成对照表回复给用户。

### 5. 结论与回写

- 用一段话回答："这个 claim 在我的复现里是否成立？偏差多少？可能的原因？"
- 询问用户是否要把复现笔记回写到知识库（`kb_add_document`）或保存为长期记忆（`save_memory`）。

## 注意事项

- **不要静默假设**：笔记里没写的超参，必须问用户或明确标注"待确认"。
- **小步快跑**：每个模块独立可跑，避免"写 500 行再 debug"。
- **保留中间产物**：checkpoint、日志、指标都落到磁盘，方便事后排查。
- **诚实汇报**：如果复现不出论文结果，如实说"未复现"，列出可能原因，不要硬凑数字。
- **资源边界**：如果论文需要 GPU / 大数据集而本机没有，提前告知用户，提出替代方案（缩小模型、subset 数据）。

## 推荐工具组合

| 阶段 | 主用工具 |
|------|---------|
| 读笔记 | `file_read` / `kb_search` |
| 澄清需求 | `ask_user_qa` |
| 规划 | `todo_add` |
| 写代码 | `file_write` / `file_edit` |
| 验证 | `run_python` |
| 回写 | `kb_add_document` / `save_memory` |
