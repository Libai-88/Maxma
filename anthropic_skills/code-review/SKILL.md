# 代码审查工作流

对用户指定的代码文件或目录做一次结构化、可执行的审查，输出分级问题清单和修复建议。

## 适用场景

- 用户说"帮我 review 这个文件 / 这个目录"
- 用户准备提交代码，希望提交前自查
- 用户怀疑某段代码有 bug / 安全问题 / 性能问题

## 工作流程

### 1. 确定审查范围

- 让用户给出绝对路径（文件或目录）。
- 若是目录，用 `file_search` 列出所有源码文件（按扩展名过滤 `.py / .js / .ts / .rs / .go / .java` 等），排除 `node_modules / .venv / dist / build / target`。
- 若文件超过 10 个，用 `ask_user_qa` 让用户聚焦到关键模块。

### 2. 逐文件审查

对每个文件用 `file_read` 读取，按以下五个维度检查：

| 维度 | 检查点举例 |
|------|----------|
| 🔴 正确性 | 空值/边界未处理、off-by-one、可变默认参数、异步竞态、异常被吞 |
| 🟠 安全 | 命令注入、路径穿越、SQL 拼接、密钥硬编码、不安全反序列化 |
| 🟡 性能 | O(n²) 循环、N+1 查询、重复 IO、无谓深拷贝、缺缓存 |
| 🟢 可维护性 | 巨函数、命名歧义、魔法数字、重复代码、缺类型注解 |
| 🔵 风格 | 与项目约定不一致（缩进、引号、命名风格） |

### 3. 分级与定位

每个问题必须包含：

```
[严重程度] 文件路径:行号 - 问题标题
  现象：……
  风险：……
  建议：……（给出可直接 `file_edit` 替换的代码片段）
```

严重程度分级：

- 🔴 **Critical**：会导致数据损坏 / 安全漏洞 / 崩溃，必须立即修
- 🟠 **High**：很可能在特定场景触发 bug，应尽快修
- 🟡 **Medium**：性能或可维护性问题，建议修
- 🟢 **Low**：风格 / 小优化，可选
- 🔵 **Nit**：纯偏好，不必修

### 4. 汇总报告

按严重程度从高到低排序，给出：

1. 总览统计（Critical X / High Y / Medium Z / …）
2. 每个问题的详细定位和建议
3. 整体评价：是否可以合并 / 发布

### 5. 可选：直接修复

- 询问用户是否要直接修 Critical / High 问题。
- 若同意，用 `file_edit` 精确替换，**一次只改一个问题**，改完用 `run_python` 跑一下相关测试（如果有）。
- 不要顺手做"清理式重构"——审查和重构混在一起会让 review 难以追溯。

## 注意事项

- **不要扫一遍就开始改代码**：审查的产出是"问题清单"，改不改由用户决定。
- **行号要准**：用 `file_read` 拿到的行号是准确的，引用时直接用。
- **建议要可执行**：不要说"建议优化一下"，要给出具体替换代码。
- **尊重项目约定**：先用 `file_search` 看看同目录其他文件的写法，不要拿别的项目的规范来要求。
- **不要鸡蛋里挑骨头**：如果没有 Critical/High，就说"没发现严重问题"，不要硬凑。
- **保密**：审查过程中读到的密钥 / 凭据，**绝对不能**回显到对话里，只提示"第 X 行发现硬编码凭据"。

## 推荐工具组合

| 阶段 | 主用工具 |
|------|---------|
| 定位文件 | `file_search` |
| 读代码 | `file_read` |
| 澄清范围 | `ask_user_qa` |
| 修复（可选）| `file_edit` / `file_manage` |
| 跑测试 | `run_python` |
| 看变更 | `git_diff` / `git_status` |
