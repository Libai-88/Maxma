# 技术文档撰写工作流

为项目生成结构化、可读性强的技术文档。**文档是给人读的**——先想读者是谁，再决定写什么、怎么写。

## 适用场景

- 用户说"给这个项目写个 README"
- 用户说"这部分代码缺文档，补一下"
- 用户说"生成 API 文档 / 架构说明 / 使用手册"

## 工作流程

### 1. 明确读者与目标

用 `ask_user_qa` 问清（一次问完）：

1. **读者**：是新人 onboarding？外部用户？代码审查者？运维？
2. **目标**：读完这份文档，读者应该能做什么？（跑起来项目 / 调用 API / 修改某模块）
3. **位置**：文档写到哪个文件？（`README.md` / `docs/api.md` / 模块内 docstring）
4. **风格**：项目里已有的文档是什么风格？先 `file_search` 找一份参照。

> 不要跳过这步。给"新人 onboarding"写的文档和给"外部用户调用 API"写的文档，结构完全不同。

### 2. 收集素材

- 用 `file_search` 列出项目结构，`file_read` 读关键文件：
  - `package.json` / `pyproject.toml` / `Cargo.toml` — 项目元信息、依赖、脚本
  - 入口文件（`main.py` / `index.ts` / `src/main.rs`）— 启动流程
  - 配置文件 — 配置项含义
  - 测试文件 — 用法示例
- 用 `git_log` 看近期变更，了解项目演进。
- 若是 API 文档，读路由定义 / 接口签名，提取参数、返回值、错误码。

### 3. 起草大纲

在动手写正文前，先把大纲发给用户确认（避免写完了才发现方向不对）：

```
# {项目名}

## 简介（1 段话）
## 快速开始（5 分钟能跑通）
## 安装
## 使用
  ### 基础用法
  ### 进阶用法
## 配置说明
## 架构概览（含目录结构）
## API 参考（如果是 API 文档）
## FAQ
## 贡献指南
## License
```

根据读者类型调整：

- **新人 onboarding**：强化"快速开始"和"架构概览"
- **外部用户**：强化"使用"和"API 参考"
- **运维**：强化"配置"和"部署"
- **代码审查者**：强化"架构概览"和"设计权衡"

### 4. 撰写正文

- 用 `file_write` 创建文件（或 `file_edit` 在已有文档上追加）。
- 每个章节遵循"一句话说明 → 代码示例 → 注意事项"三段式。
- 代码示例必须**可复制即用**——不要写伪代码、不要省略 import。
- 配置项用表格列出：`名称 | 类型 | 默认值 | 说明`。

### 5. 自检与校对

写完后自己读一遍，检查：

- [ ] 快速开始里的命令真的能跑通吗？（用 `run_python` 或 `file_search` 验证一下）
- [ ] 代码示例有没有漏 import？
- [ ] 配置项表格里有没有过时的字段？（对照实际配置文件 `file_read`）
- [ ] 目录结构和实际磁盘结构一致吗？（用 `file_search` 再核一遍）
- [ ] 有没有"TODO"、"待补充"这种占位符没填？

### 6. 交付

- 把文档路径告诉用户，让用户 review。
- 询问是否需要同步更新其他文档（如 `CHANGELOG.md` / 项目主页）。

## 文档风格规范

- **标题层级**：`#` 只用于文档标题，`##` 用于章节，`###` 用于子节。不要跳级。
- **代码块**：必须标语言（` ```python ` / ` ```bash `）。
- **链接**：用相对路径链接项目内其他文档，不要用绝对路径。
- **emoji**：除非项目已有风格，否则不用。专业文档里 emoji 显得轻浮。
- **长度**：单段不超过 5 行。超过就拆。

## 注意事项

- **不要写"显然"、"当然"、"简单地说"**：对读者来说不显然。
- **不要照抄代码注释当文档**：注释是写给改代码的人看的，文档是写给用代码的人看的。
- **不要写过期的"截止 2024 年"**：文档里别写时间点，否则很快过期。
- **不要用 `file_edit` 大段重写已有文档**：会丢失历史。新增章节追加，修改章节小范围 edit。
- **保密**：从代码里读到的密钥、内部 URL、内部 API，**绝对不能**写进文档。

## 推荐工具组合

| 阶段 | 主用工具 |
|------|---------|
| 澄清读者 | `ask_user_qa` |
| 看项目结构 | `file_search` |
| 读代码 | `file_read` |
| 看历史 | `git_log` |
| 写文档 | `file_write` / `file_edit` |
| 验证示例 | `run_python` |
