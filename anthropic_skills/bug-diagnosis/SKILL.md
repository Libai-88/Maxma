# 问题诊断工作流

把"用户描述的模糊症状"转化成"已定位的根因 + 可执行的修复"。**不要看到描述就猜原因开方子**，必须走完闭环。

## 适用场景

- 用户报告："知识库搜不到"、"上传后没反应"、"连接失败"、"突然报错了"
- 某个内置工具 / 工作流跑不起来
- 行为与预期不符但没明确报错

## 工作流程

### 1. 复现并固定症状

- 用 `ask_user_qa` 问清三件事（不要一次问太多）：
  1. **操作步骤**：你做了什么？（精确到点了哪个按钮 / 调了哪个工具）
  2. **预期 vs 实际**：你期望看到什么？实际看到什么？
  3. **时机**：是第一次就出问题，还是之前能用、现在突然不行？
- 如果有报错信息，让用户原样贴出来（不要让用户"凭印象描述"错误）。
- 如果是工具调用失败，调一次同样的工具参数，亲眼看到错误。

> 关键：**先看到症状，再下假设**。不要在没复现的情况下就开始改代码。

### 2. 收集症状证据

按"由近及远"顺序收集：

1. **当下日志 / 报错**：用 `file_search` 找日志文件（Maxma 日志在 `%APPDATA%/MaxmaHere/logs/`），用 `file_read` 读最近一段。
2. **配置 / 状态**：检查相关配置文件（`mcp_servers.yaml` / `env` / `active_persona.yaml`）。
3. **数据层**：如果是 KB / 记忆问题，用 `kb_list_documents` / `list_memories` 看实际数据状态。
4. **代码层**：用 `file_search` + `file_read` 看相关模块实现。
5. **历史记忆**：用 `search_memories` 搜一下"以前有没有遇到过类似问题"。

每收集一条证据，在回复里**显式列出**，让用户看到推理过程。

### 3. 提出根因假设

基于证据列出 2–3 个候选根因，按可能性排序：

```
假设 A（最可能，60%）：[根因描述]
  支持证据：……
  反证：……
假设 B（30%）：……
假设 C（10%）：……
```

### 4. 验证假设

对最可能的假设设计**最小验证实验**：

- 如果是代码 bug：用 `run_python` 写一段最小复现脚本，验证 bug 确实存在。
- 如果是配置问题：临时改配置（用 `file_edit`），重试一次，看是否解决。
- 如果是数据问题：直接查数据（如 `kb_search` 看是否真的没索引）。

**验证通过才下结论**。验证失败就回到第 3 步换下一个假设。

### 5. 给出修复并确认

- 验证通过后，给出具体修复方案（`file_edit` 改哪一行 / 配置怎么调 / 数据怎么补）。
- 修复后**重跑一次原始操作**，确认症状消失。
- 把根因和修复方案存到长期记忆（`save_memory`），下次遇到类似问题可以快速定位。

## 常见根因清单（按 Maxma 实际经验）

| 症状 | 高频根因 | 验证方法 |
|------|---------|---------|
| KB 搜不到内容 | embedding/向量库初始化失败但被吞 | `kb_search` 看返回的 `warning` 字段；查日志 `[rag]` / `[kb]` |
| MCP 工具不出现 | 配置 YAML 用了 `servers:` 而非 `mcp_servers:` | `file_read` 看 `mcp_servers.yaml` 顶层 key |
| 工具调用报"路径不在白名单" | 路径不在 `path_whitelist` 或撞到 MaxmaBlocker | `manage_whitelist list` 看白名单 |
| 人格行为规则丢失 | 打包模式下 `AGENTS.md` 未复制到 `PERSONAS_DATA_DIR` | `file_search` 在 `%APPDATA%/MaxmaHere/config/personas/` 找 `AGENTS.md` |
| 子进程残留 | 父进程监控失效（`os.getppid` 返回 bootloader） | 看是否有 `maxma-server.exe` 在 Tauri 退出后仍存活 |
| Python 沙箱首次调用慢 5–10s | 打包模式下 `sys.executable` 指向 bootloader，每次触发解压 | 看日志首次调用耗时 |

## 注意事项

- **不要跳步**：复现 → 证据 → 假设 → 验证，缺一不可。跳步等于"猜"。
- **诚实汇报不确定**：如果验证不充分，明说"目前怀疑是 X，但还没验证"，不要把猜测说成结论。
- **不要边查边改**：诊断阶段只读不写，根因确认后再统一改。
- **保留现场**：日志 / 配置 / 数据快照在修复前先备份（`file_manage` 复制一份），万一改坏了能回滚。
- **闭环**：症状消失 ≠ 修复完成。要能解释"为什么之前不工作、为什么改了就好了"，才算闭环。

## 推荐工具组合

| 阶段 | 主用工具 |
|------|---------|
| 澄清症状 | `ask_user_qa` |
| 读日志 | `file_search` / `file_read` |
| 看配置 | `file_read` |
| 查数据 | `kb_search` / `kb_list_documents` / `list_memories` |
| 查历史 | `search_memories` |
| 验证假设 | `run_python` / `file_edit` |
| 修复 | `file_edit` / `manage_whitelist` |
| 回写经验 | `save_memory` |
