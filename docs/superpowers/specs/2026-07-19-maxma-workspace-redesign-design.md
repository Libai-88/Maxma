# Maxma 工作区前端重构设计规格

> 日期：2026-07-19
> 状态：已确认的设计基线
> 适用范围：`MaxmaHere/web/src` 的工作区壳层与主对话页
> 约束：本规格只定义信息架构、视觉和实施边界，不改变后端协议或现有聊天功能

## 1. 背景与问题

当前主界面已经具备完整的聊天、会话、模型、工具、工作台和表情系统，但多个层级同时展示相同或相近的信息：

- 侧边栏同时承担全局导航、会话管理、模型设置、工具入口和健康信息，内容过密。
- 顶部栏和输入框都出现模型、上下文或状态信息，用户难以判断哪个控件是当前生效入口。
- 顶部存在与核心对话无直接关系的预留或弱使用频率控件，压缩了可用空间。
- 工作台能力已经存在，但不应默认占据聊天主区；需要按需打开并在关闭时完全让出空间。
- 现有布局在不同视口下会出现内容被截断、输入区被挤出、侧栏信息无法完整阅读的问题。

本次重构采用已确认的混合方案：吸收 `D:\Maxma\docs\codex-redesign` 和 `D:\Maxma\dist` 中 Dawn 方向的层次、暖色和工作台思路，同时保留 Maxma 的品牌、现有功能和 Vue 数据流。

## 2. 目标与非目标

### 2.1 目标

1. 让主工作区的第一视觉焦点回到“当前会话、对话流、输入框”。
2. 将全局导航、会话管理、聊天控制和工作台职责分开，减少同一功能的重复入口。
3. 在 `390px`、`768px`、`1280px` 三类视口上保持可用，不依赖用户调整浏览器缩放倍率。
4. 保持当前聊天协议、会话生命周期、模型配置、流式回复、工具调用、表情包和工作台数据不变。
5. 让高频操作保持可见，让低频或上下文相关操作按需展开。
6. 建立可执行的视觉、可访问性和回归验收标准，为后续分阶段实现提供边界。

### 2.2 非目标

- 不重写 FastAPI、WebSocket、provider、agent 或工具执行逻辑。
- 不改变 REST endpoint、WebSocket event type、payload 结构、会话 ID 或本地存储键。
- 不新增模型路由、provider 能力、记忆算法、工作流引擎或 Agent 功能。
- 不把 `dist` 中的静态 HTML 直接替换为生产页面；它只作为视觉和布局参考。
- 不照搬 Codex 的品牌标识、角色插画、产品文案或固定四栏尺寸。
- 不在本阶段重做所有设置页、活动页、动态页和管理页面；它们只需要适配新的全局壳层入口。
- 不删除现有功能代码。对主界面暂时不适合常驻展示的功能，优先改为抽屉、菜单或既有页面入口。
- 不以装饰性背景、复杂动画或字体加载作为功能完成条件。

## 3. 核心信息架构

目标结构如下：

```text
App.vue
├── IconRail                         全局图标栏，始终存在
├── SessionDrawer                    会话抽屉，按需展开
│   ├── 新建会话
│   ├── 固定/临时会话列表
│   └── 会话级操作
├── MainWorkspace
│   ├── WorkspaceHeader              当前会话和少量有效状态
│   └── ChatView
│       ├── ConversationStream        对话、思考、工具调用、表情包
│       ├── WorkflowInline            有数据时才出现的工作流区域
│       ├── Composer                   单一模型入口和消息输入
│       └── WorkbenchDrawer            按需打开的输出/画布/推理工作台
└── GlobalOverlays                    toast、媒体查看器、引导、错误边界
```

### 3.1 图标栏 Icon Rail

**职责**：提供稳定的全局导航锚点和抽屉开关，不承载复杂内容。

**默认内容**：

- Maxma 品牌图标或简化标识。
- 对话入口。
- 活动/动态入口，沿用现有路由。
- 设置入口，指向现有设置相关路由或菜单。
- 会话抽屉开关。
- 底部保留帮助或版本入口时，必须有明确的实际页面；无功能的占位入口不显示。

**行为**：

- 桌面端默认显示图标栏，宽度稳定，不因标签文本长度变化。
- 悬停可显示 tooltip，键盘焦点时也必须显示可理解的名称。
- 点击会话图标打开或关闭 `SessionDrawer`，不改变当前会话。
- 当前路由使用明显的 active 状态；active 状态不能只依赖颜色。
- 在窄屏上保留图标栏，但图标按钮扩大到不小于 `44px` 触控区域。

### 3.2 会话抽屉 Session Drawer

**职责**：集中处理会话浏览和会话级操作，替代当前把会话内容长期铺满侧栏的做法。

**内容层级**：

1. 抽屉头部：标题“会话”、新建会话按钮、关闭按钮。
2. 会话列表：固定会话优先，临时会话其次；显示名称、相对时间、消息数量或状态点。
3. 会话操作：切换、固定/取消固定、重命名、删除，沿用现有右键菜单或显式菜单。
4. 空状态：只说明没有保存会话，并提供新建入口；不显示与操作无关的长说明。

**不再常驻的内容**：模型参数、完整工具清单、健康面板和冗长的会话操作说明。它们保留原有功能，通过设置页、工具入口、状态详情或工作台按需访问。

**行为**：

- 打开抽屉不卸载聊天主区，不重新创建 WebSocket。
- 切换会话继续调用现有 `sessionStore.switchSession` 流程，并回到 `/`。
- 创建、删除、固定和取消固定继续使用现有 store action 与确认逻辑。
- 抽屉打开时，焦点进入抽屉；关闭时返回触发按钮；按 `Escape` 关闭。
- 移动端抽屉使用覆盖层，点击遮罩关闭；桌面端可采用侧滑或相邻面板，但宽度必须受约束。

### 3.3 顶部栏 Workspace Header

**职责**：只表达当前工作上下文，不重复输入区的模型选择。

**保留信息**：

- Maxma 或当前 Agent 名称的短版。
- 当前会话标题或可读的临时会话标识。
- 连接状态：已连接、连接中、断开、错误。
- 工作台开关。
- 一个更多操作入口，用于低频且确实可用的会话设置。

**默认移出顶部栏**：

- 第二个模型选择器。
- 常驻上下文用量详情。
- 没有实际动作的预留任务按钮。
- 过长的人设、场景或系统提示全文。

**控制归属原则**：

- 模型由输入区的唯一 `ModelSelector` 管理。
- 上下文用量由输入区同一行的紧凑 `ContextUsageBadge` 表示，详情使用 tooltip 或弹层。
- 私密模式、自动执行、会话权限等已有有效开关可以保留，但默认收纳到“更多”或会话级菜单；若某控件当前没有真实行为，不在主界面渲染。
- 顶部栏不能因为长会话标题、中文人设或状态文案而撑破布局，标题允许截断并提供完整 tooltip。

### 3.4 对话区 Conversation Stream

**职责**：展示用户消息、Agent 回复、流式思考、工具调用、错误、引用和表情包。

**布局原则**：

- 对话区是主区域，使用 `flex: 1` 和 `min-height: 0`，输入区必须始终拥有稳定空间。
- 消息列表内部滚动；页面根节点不因消息增长产生横向滚动。
- 保持用户消息、Agent 消息、系统提示和工具调用的现有语义区别。
- 工具调用可折叠，默认展示状态和必要摘要；详细输出在消息内展开或转入工作台。
- `[表情:...]`、`[表情包:...]` 和已有 sticker 数据继续由现有解析与渲染链路处理，视觉重构不得改成纯文本显示。
- 流式输出、自动滚动、手动滚动后不抢焦点等现有行为必须保持。
- 空会话显示 `WelcomeScreen`，但快速开始项目必须是真实可执行的发送动作，不使用无效装饰卡片。

### 3.5 输入区 Composer

**职责**：完成消息编辑、引用/附件/技能选择、表情选择、唯一模型选择和发送/停止。

**推荐层级**：

```text
Composer
├── 上下文/引用/附件预览（有内容才显示）
├── 多行消息输入
└── 操作栏
    ├── 添加、表情、引用或技能入口
    ├── 唯一 ModelSelector
    ├── 紧凑 ContextUsageBadge
    └── 发送或停止按钮
```

**交互约束**：

- 保持 Enter 发送、Shift+Enter 换行；输入法组合期间不得误发送。
- 发送中显示停止操作，停止继续调用现有取消流程。
- 模型下拉只出现一次，切换继续写入现有 provider/chat store。
- 保持文件、文件夹、图片、引用、技能、宏、表情包等已有入口；低频入口可通过 `+` 菜单聚合，但不能改变其 payload。
- 输入框自动撑高有上限；超过上限后内部滚动，不能推动页面头部或工作台出界。
- 禁用、加载、错误、空文本和连接失败状态均有明确反馈。

### 3.6 按需工作台 Workbench Drawer

**职责**：展示工具结果、推理时间线、画布卡片、文件/代码/表格/摘要以及其他已实现的辅助产物。

**默认状态**：关闭，不占用主对话区宽度；首次有可查看内容时可以提示入口，但不能强制打开遮挡消息。

**内容映射**：

- `ReasoningTimeline`：推理或事件时间线。
- `CanvasContainer` / `CanvasTabs`：可交互画布和卡片页签。
- 现有 workbench cards：代码、表格、摘要、确认、选择等结果。
- 工具输出、浏览器或来源视图只有在有对应数据和已实现交互时才显示。

**行为**：

- `WorkbenchPanel` 的 open/close、active tab、card count 和 remove 事件保持不变。
- 桌面端优先使用右侧抽屉；中等视口使用覆盖式面板；移动端使用全屏或底部 sheet。
- 工作台打开时主区必须仍可滚动，不能同时出现两个不可见的滚动容器。
- 关闭工作台不删除卡片或重置 active tab；重新打开恢复当前状态。
- 大型或高开销内容按需挂载，关闭时不影响聊天流。

## 5. 数据与交互不变约束

这是视觉重构，不是协议重构。实现时必须满足以下不变量：

### 5.1 通信与状态

- 继续使用当前 REST API、Vite 代理和 WebSocket 地址解析方式。
- `useChat.ts` 对 `thinking_start`、`token`、`thinking_end`、`tool_start`、`tool_end`、`tool_error`、`answer`、`done`、`error` 等事件的含义不变。
- 连接、重连、取消、错误分类和 trace id 展示行为不变。
- provider、模型、temperature、max tokens、thinking、context usage 的数据来源和写入方式不变。
- 会话创建、切换、删除、固定、取消固定、重命名和历史恢复不变。
- 私密模式、自动执行和会话权限若保留入口，必须继续调用原有 action；若仅隐藏入口，不得删除底层状态。

### 5.2 消息与 Agent 产物

- 用户消息的文本、引用、文件/文件夹/图片引用、技能、宏和 sticker 标记不能被视觉层改写。
- Agent 回复的 Markdown、代码块、图片、来源、工具卡片和表情包必须继续渲染。
- `[表情:名称]`、`[表情包:名称]` 与既有 sticker path/URL 的解析结果不变；失败时仍按既有降级策略处理。
- 工作流、子 Agent、计划确认、工具 action、卡片 pin/remove 和媒体查看行为不变。
- 关闭抽屉或工作台不应清空消息、卡片、滚动位置、当前模型或会话连接。

### 5.3 入口唯一性

- 主工作区中只保留一个可见模型选择入口。
- 上下文用量只保留一个紧凑摘要入口，详情可以通过 tooltip、popover 或工作台查看。
- 一个动作只能有一个主按钮；同一动作的快捷方式可以存在，但不能制造两个语义相同的常驻按钮。
- 无后端或无状态支持的控件不显示为“可用”状态，不使用灰色空壳占据布局。

## 6. 响应式规格

采用固定断点配合流式宽度。所有断点均以 CSS 容器约束为准，不能依赖浏览器缩放。

### 6.1 `390px`：移动窄屏

- 图标栏保留，宽度 `56px` 左右；不得显示长标签。
- 会话抽屉为覆盖式全高面板，最大宽度不超过视口，打开时有遮罩和焦点陷阱。
- 顶部栏只显示会话短标题、状态图标/短文字、工作台入口和更多按钮。
- 对话区占据剩余高度，消息气泡和工具卡片宽度使用 `min(100% - 24px, 720px)` 类似约束，不出现横向滚动。
- 输入区固定在对话区底部，操作栏允许换行；模型选择器可缩短为当前模型名，但仍只出现一次。
- 表情、附件、引用、发送/停止按钮触控区域不小于 `44px`。
- 工作台使用全屏或底部 sheet；关闭后恢复对话滚动位置。
- 长中文标题、模型名、错误文案必须换行或省略，不能遮挡按钮。

### 6.2 `768px`：平板/窄桌面

- 图标栏固定；会话抽屉可使用 `280px` 左右的侧滑面板，不压缩到不可读。
- 顶部栏保留会话标题、状态、工作台和更多入口；隐藏非核心详情。
- 对话主列使用最大宽度约束并保持左右内边距，输入区操作栏优先单行，空间不足时仅允许有序换行。
- 工作台打开时采用右侧覆盖面板或受限列宽；对话和输入区仍可用。
- 页面、主列、消息列表和工作台分别设置 `min-width: 0`、`min-height: 0`；只允许预期容器滚动。

### 6.3 `1280px` 及以上：桌面

- 图标栏常驻，默认宽度稳定；会话抽屉按需展开，建议宽度 `280-320px`。
- 主对话列使用 `minmax(0, 1fr)`，内容最大宽度由可读性约束控制，不强行铺满超宽屏。
- 工作台打开时显示右侧抽屉，建议 `320-420px`，关闭时完全释放空间。
- 顶部栏控件全部在同一行，但不放置第二模型选择器或长人设文本。
- 输入区在单行操作栏中显示唯一模型选择器、上下文摘要和发送/停止；文本输入可多行撑高但有最大高度。
- 若采用背景纹理或装饰图，装饰层不参与布局、不可拦截点击，且不得影响文字对比度。

### 6.4 通用布局不变量

- `html`、`body`、`#app`、壳层和主列必须支持 `min-width: 0`，根页面禁止意外横向溢出。
- 任何固定宽度控件都必须有最大宽度、溢出策略和窄屏变体。
- 不使用随 viewport 宽度连续缩放字体的方案；字号按 token 和断点离散调整。
- 输入区、工作台和抽屉不能依靠负 margin 互相覆盖。
- 浏览器缩放倍率为 100% 时，三类视口均可完成发送消息、切换会话、打开/关闭工作台。

## 7. 视觉 Token 与品牌原则

### 7.1 Dawn 方向的吸收范围

从 `codex-redesign` 的 Dawn 方向吸收：

- 温暖的米杏/奶油背景、低饱和玫瑰金强调和轻量古金点缀。
- 通过浅色层、细边界和轻阴影区分层级，而不是堆叠厚重卡片。
- 少量水彩/纸张氛围作为背景层，不能覆盖正文或抢夺焦点。
- 直角或小圆角的工具型容器，避免每个区域都变成圆润卡片。
- 语义清晰的状态色：连接成功、警告、错误、信息必须可区分。

### 7.2 Maxma 品牌保留

- 品牌名、logo、Maxma 人设和中文产品文案保持 Maxma 语境，不使用 `Atelier Agent` 作为界面品牌。
- 现有主题系统和 `tokens.css`/theme 文件是实现入口；不得在每个组件内复制一套 Dawn 变量。
- Maxma 的聊天人格可以在详情或消息中体现，但顶部只显示短版，不让长人设成为布局负担。
- 中文正文优先使用当前可读的中文字体栈；装饰性西文字体只用于品牌或极少量标题，不影响中文信息扫描。
- 代码、路径、命令、模型 ID 使用等宽字体；中文正文保证稳定字宽、足够行高和清晰字重。

### 7.3 Token 使用规则

- 颜色、间距、圆角、阴影、字号、z-index 和动效时长优先引用现有主题 token；缺少 token 时先在设计规格中定义，再实现。
- Dawn 的 `#F2E6DD`、`#FAF1E8`、`#D4A5A5`、`#8B5A6B`、`#C9A876` 等只能作为参考色，不得无视 Maxma 当前主题和高对比主题。
- 普通正文和按钮文字必须达到 WCAG AA；原设计中低对比度 muted 色只能用于非关键信息，不得作为唯一状态表达。
- 主按钮、发送按钮和用户消息气泡需单独验证文字对比度；不能因为暖色背景而牺牲可读性。
- 装饰元素最多作为背景辅助，不得使用大面积渐变、离散光晕或无意义的装饰球体填充空白。
- 阴影保持轻量；层级优先由结构、边界、留白和 active 状态表达。
- 圆角采用小范围、功能化原则：工具容器和 composer 可使用小圆角，导航项、消息内容和列表不应全部变成胶囊。
- 所有图标按钮使用项目现有图标组件或同等可识别图标；不以 emoji 代替核心 UI 图标。
- `prefers-reduced-motion: reduce` 下禁用非必要位移、闪烁和装饰过渡。

## 8. 可访问性要求

- 使用 `nav`、`main`、`header`、`aside`、`section`、`form` 等语义元素表达区域。
- 图标栏、会话抽屉、工作台、模型选择器、更多菜单和表情选择器必须有可读的 `aria-label` 或可见名称。
- 抽屉和弹层使用 `aria-expanded`、`aria-controls`、`role="dialog"` 等正确关系；打开后管理焦点，关闭后恢复触发元素焦点。
- 键盘必须能完成：打开会话抽屉、切换会话、关闭抽屉、选择模型、发送/停止、打开工作台、切换 tab、关闭弹层。
- `Escape` 只关闭当前最高层弹层，不应意外中断聊天或清空输入。
- active、连接、错误、运行中等状态不能只通过颜色表达，应同时有文字、图标、形状或辅助技术可读信息。
- focus-visible 外环必须在浅色 Dawn 和现有深色/高对比主题中可见，不能被 `overflow: hidden` 裁剪。
- 输入框、按钮、select、slider 和菜单项的可点击区域不小于 `44x44px`（桌面紧凑图标按钮可有视觉尺寸，但命中区域不能过小）。
- 中文正文默认保持足够行高；长文案、错误信息和模型名允许自然换行，不使用负字间距压缩中文。
- 不以 hover 作为唯一交互方式；移动端和键盘用户必须有等效入口。
- 尊重 `prefers-reduced-motion`，并为高对比主题保留明确边界和状态。

## 9. 测试与验收标准

### 9.1 结构和视觉验收

- 在 `390x844`、`768x900`、`1280x800` 至少各运行一次浏览器验收；推荐另加 `1440x900` 桌面检查。
- 三类视口均无意外横向滚动，输入区、发送/停止按钮和最后一条消息可见。
- 默认页面只出现一个模型选择器；顶部栏、输入区和工作台不重复展示同一状态摘要。
- 图标栏、会话抽屉、顶部栏、对话区、输入区和工作台的边界清晰，任何文本不覆盖相邻控件。
- 会话抽屉打开/关闭、工作台打开/关闭、窗口缩放和浏览器刷新后，主对话内容不丢失。
- 空会话、有历史消息、流式回复、工具调用、错误消息、表情包消息和工作流消息均不出现空壳高度或布局跳动。
- 长标题、长模型名、长中文错误和宽代码块可读且不撑破父容器。

### 9.2 功能回归验收

- 创建、切换、删除、固定、取消固定、重命名会话仍可用。
- provider 加载成功、失败、重试和无 provider 引导仍可用。
- 模型切换仍更新当前会话使用的 provider/model；发送消息后 payload 与重构前一致。
- Enter 发送、Shift+Enter 换行、输入法组合、停止生成和重新连接行为不变。
- 流式 token、思考块、工具开始/完成/错误、最终回答和 done 状态正常渲染。
- Agent 回复中的 `[表情:...]`、`[表情包:...]` 和现有 sticker 图片仍能嵌入；图片加载失败按原逻辑降级。
- 引用、文件/图片/文件夹引用、技能/宏、附件预览、表情选择器和相关发送 payload 不变。
- 工作台 tab、推理时间线、canvas 卡片、卡片移除、artifact action 和媒体查看仍可用。
- 私密模式、自动执行、权限控制和上下文摘要若仍暴露入口，点击后实际状态必须改变并反馈给用户。

### 9.3 可访问性验收

- 只用键盘可以从图标栏进入当前会话、打开抽屉、切换模型、输入并发送消息、打开工作台并关闭。
- 使用屏幕阅读器或 accessibility tree 能识别区域名称、按钮名称、连接状态、模型名称和消息角色。
- 通过自动化检查确认没有未关联 label 的表单控件、重复 id、不可达焦点或 aria 状态错误。
- Dawn、默认主题、暗色主题和高对比主题下，正文、按钮、状态和 focus ring 达到可读标准。

### 9.4 自动化建议

- 保留并通过现有单元测试、环境/代理测试、表情解析测试和后端 provider 测试。
- 为布局增加 Playwright 视口测试：检查 `document.documentElement.scrollWidth <= window.innerWidth`、关键元素 bounding box、抽屉焦点和模型选择器数量。
- 为消息渲染增加回归样例：Markdown、代码、工具调用、`[表情包:开心]`、图片和流式中间态。
- 为会话和工作台添加最小交互 smoke test，不要求重写现有 store 测试。
- 视觉快照只锁定结构、溢出和关键状态，不锁定可能随字体渲染变化的整页像素。

## 10. 分阶段实施边界

### Phase 0：基线与拆分清单

**范围**：确认现有路由、store、事件、props/emits 和关键视口截图，建立重构前基线。

**允许**：增加测试或记录，不改变运行时行为。

**禁止**：顺手修复后端、改协议、删除旧组件功能或改变主题数据。

### Phase 1：壳层与信息架构

**范围**：从 `App.vue` 和侧栏开始拆分图标栏、会话抽屉、主工作区容器；接通现有 `useSidebar` 和 `sessionStore`。

**验收重点**：导航、抽屉打开/关闭、会话 CRUD、路由切换和焦点行为不变；没有第二份会话列表。

**不包含**：新的数据能力、设置页重写、工作台卡片重写。

### Phase 2：顶部栏与 Composer 归属

**范围**：精简 `ChatHeader.vue`，把 `ModelSelector` 和 `ContextUsageBadge` 的可见归属统一到 `ChatInput.vue`，收纳低频控制。

**验收重点**：模型只有一个可见入口，输入/发送/停止/表情/附件/引用/技能/宏行为不变；长文本在三类视口不溢出。

**不包含**：provider store、WS payload、输入逻辑或模型选择逻辑重写。

### Phase 3：按需工作台

**范围**：将现有 `WorkbenchPanel`、`CanvasContainer`、`CanvasTabs`、`ReasoningTimeline` 和 cards 组织为抽屉/覆盖面板。

**验收重点**：默认关闭不占空间，打开可查看已有产物，关闭不丢卡片、不重置状态、不影响聊天滚动和 WebSocket。

**不包含**：新增工具、canvas 类型、后台进程能力或浏览器能力。

### Phase 4：视觉收敛、响应式与可访问性

**范围**：将 Dawn 方向映射到 Maxma theme/token，完成 `390/768/1280` 断点、焦点管理、对比度、动效降级和布局回归测试。

**验收重点**：达到第 9 节全部验收标准；清除空壳入口、重复模型入口和不必要的常驻信息。

**不包含**：扩展产品功能、主题大迁移、后端重构或全站页面重做。

### Phase 5：旧壳层清理

**前置条件**：Phase 1-4 的回归通过，并确认无路由、事件或用户入口依赖旧结构。

**范围**：删除已确认无引用的旧布局样式和重复模板；保留兼容组件，直到所有调用点迁移完成。

**禁止**：以“看起来不用了”为理由直接删除尚未验证的 store action、composable、事件处理或设置入口。

## 11. 完成定义

本设计完成并可进入实现的条件是：

1. 任何主界面控件都能明确归属于图标栏、会话抽屉、顶部栏、对话区、输入区或工作台之一。
2. 主界面默认只展示当前会话、连接状态、对话内容、输入和一个模型入口。
3. 所有现有功能的状态源、通信协议、消息渲染和工作台数据契约均有明确保留说明。
4. `390px`、`768px`、`1280px` 的布局和交互验收标准可由浏览器自动化执行。
5. Dawn 只提供视觉方向，Maxma 品牌和中文可读性优先级更高。
6. 后续实现可以按 Phase 0-5 独立提交和回滚，每阶段都有清晰的行为边界。
