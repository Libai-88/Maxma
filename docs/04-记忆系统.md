# 记忆系统

## 四层记忆架构

MaxmaHere 的记忆系统分为四层：

```
┌─────────────────────────────────────────────┐
│  短期记忆 (ShortTerm) — Graph Checkpointer     │
│  - 当前会话的消息历史                          │
│  - LangGraph 内置，消息随会话累积               │
│  - 1M 上下文窗口，不做 token 裁剪              │
├─────────────────────────────────────────────┤
│  长期记忆 (LongTerm) — YAML + 向量检索          │
│  - 跨会话的自然语言记忆（用户画像/偏好/事件）    │
│  - 每轮对话后 LLM 增量更新                      │
│  - 存储介质：config/personas/MEMORY.md         │
├─────────────────────────────────────────────┤
│  情景记忆 (Episodic) — JSON + 向量检索          │
│  - 对话快照库，按 turn 存储摘要                 │
│  - 支持"上次我们聊到哪了"式回忆                 │
│  - 存储介质：data/episodic_memory.json         │
├─────────────────────────────────────────────┤
│  语义记忆 (Semantic) — JSON + 向量检索          │
│  - 结构化事实三元组（subject-predicate-object）│
│  - 从对话/文档中抽取，去重存储                  │
│  - 存储介质：data/semantic_memory.json         │
└─────────────────────────────────────────────┘
```

---

## 第一层：短期记忆（Graph Checkpointer）

会话消息由 LangGraph 的 `MemorySaver` checkpointer 管理，以 `thread_id` 为键存储每轮对话中 Agent 图产生的完整消息列表（含 HumanMessage、AIMessage、ToolMessage）。外部通过 `checkpointer.aget_state()` 异步读取。

`SessionState` 上仅保留 `message_count` 计数器供同步场景（如列表页）快速获取消息数量，实际消息内容统一走 checkpointer。

---

## 第二层：长期记忆（Memory Narrative）

在 [memory/narrative.py](../memory/narrative.py) 中。

### 工作原理

每轮对话结束后，将本轮**裸对话消息**（不做任何预处理）连同旧的 `MEMORY.md` 一起送给 LLM，让 LLM 在现有记忆基础上融合新信息，写回 `config/personas/MEMORY.md`。

```
回合结束:
  turn_messages (裸消息)
      │
      └── update_narrative(llm, turn_messages)
            │
            ├── 冷启动 (MEMORY.md 不存在):
            │     turn_messages → LLM → 生成 MEMORY.md
            │
            └── 常态 (MEMORY.md 已存在):
                  turn_messages + 旧 MEMORY.md → LLM → 更新 MEMORY.md
```

### 为什么用裸消息而不是提取的摘要？

旧系统先用 Extractor 把对话压缩成 summary，再拿 summary 去检索或生成。这导致信息损失叠加，且两个模块耦合。叙事模块直接读裸消息，让 LLM 看到原始对话的全部细节。

### 叙事 LLM Prompt

LLM 通过四个 CRUD 工具（`create_memory` / `read_memories` / `update_memory` / `delete_memory`）管理记忆条目，而非直接撰写文本。

**冷启动**（MEMORY.md 不存在时）：LLM 从空状态开始，用 `create_memory` 逐条添加事实，自行选择合适的分区名。

**常态更新**（MEMORY.md 已存在时）：LLM 先调用 `read_memories` 查看现有条目（每条带唯一 ID 和分区），对比新对话后增/改/删条目。系统提示词会列出建议分区名作为参考，但 LLM 在现有分区不合适时可自行新建（1-4字中文名词）。

**分区建议**（系统提示词中提供）：
- "身份" — 基本身份信息，极少变化
- "音乐" — 虚拟歌手、声库、歌曲、专辑、创作者
- "品味" — 电影、美食、UP主、品牌偏好等
- "地点与路径" — 具体地点和文件系统路径
- "瞬间" — 即时观察和感受，短暂轮替
- "时效待办" — 有截止日期的事项，到期后应删除

### 叙事注入系统提示词

在 [agent/prompts.py](../agent/prompts.py) 中，`build_system_prompt()` 将 `MEMORY.md` 的内容注入到 `"## 我对用户的记忆"` 段落。每轮对话直接调用 `build_system_prompt()` 获取最新叙事。

### 叙事内容示例（分区格式）

```markdown
# 长期记忆索引
- [身份](#身份)
- [音乐](#音乐)
- [瞬间](#瞬间)

---

## 身份
- 用户就读于北京邮电大学沙河校区，从事网络空间安全领域工作。
- 用户是广东人，家乡在江门市蓬江区。

## 音乐
- 用户特别喜欢洛天依。
- 用户最喜欢的歌曲是洛天依的《海边城》。

## 瞬间
- 用户正在吃汉堡王的外卖（芝士牛堡）。
```

### 文件存储

| 文件 | 说明 |
|------|------|
| `config/personas/MEMORY.md` | AI 自动生成的叙事，每轮更新，gitignore |
| `config/personas/USER.md` | 用户手动填写的自述，gitignore |
| `config/personas/USER.example.md` | 用户自述模板，git 跟踪 |
| `memory/user_init.py` | 首次运行从 example 复制 USER.md |

## 第三层：情景记忆（Episodic Memory）

在 [memory/episodic.py](../memory/episodic.py) 中。

### 工作原理

每轮对话结束后，`commit_to_episodic()` 从当前 checkpoint 提取对话摘要，生成一个 episode 快照（含 turn_id、timestamp、摘要文本、消息数），写入 JSON 文件和 chromadb `episodic` collection。

```
回合结束:
  checkpoint 对话消息
      │
      └── commit_to_episodic(graph, config, episodic_mm)
            │
            ├── LLM 摘要（可用时）或文本提取回退
            │
            └── episodic_mm.add_episode(summary, session_id, turn_id)
                  ├── 写入 data/episodic_memory.json
                  └── 索引到 chromadb episodic collection
```

### 情景检索注入

在 [agent/graph.py](../agent/graph.py) 中，`episodic_retriever_node` 位于 `planner` 和 `agent` 之间：以最新 HumanMessage 作为查询，调用 `retrieve_from_episodic()` 向量检索相关历史 episode，注入到 `AgentState.episodic_context`，再由 `agent_node` 拼接到系统提示词。

### 支持"上次我们聊到哪了"式回忆

Agent 也可通过 `search_episodic` 工具主动检索历史对话场景，返回 `[timestamp] (相似度 X%) summary` 格式的结果。

---

## 第四层：语义记忆（Semantic Memory）

在 [memory/semantic.py](../memory/semantic.py) 中。

### 工作原理

语义记忆存储结构化事实三元组（subject-predicate-object），如「用户 喜欢 古典音乐」。`add_fact()` 支持去重（相同三元组只保留一条，更新 timestamp）。数据写入 JSON 文件和 chromadb `semantic` collection。

### 语义事实注入系统提示词

在 [agent/prompts.py](../agent/prompts.py) 中，`_scan_semantic_facts()` 扫描所有语义记忆事实，格式化为 `## 语义记忆（结构化事实）` 段落注入系统提示词。语义记忆变更时通过 `invalidate_prompt_cache()` 失效缓存。

### 跨层工具分发

记忆工具（read/list/delete/update）按 ID 前缀自动识别层级：
- `ep_*` → 情景记忆层
- `fact_*` → 语义记忆层
- 其他 → 长期记忆层

`create_memory` 通过 `layer` 参数显式指定层级（long/episodic/semantic）。

---

## 四层协调器（MemoryCoordinator）

在 [memory/coordinator.py](../memory/coordinator.py) 中。

`MemoryCoordinator` 提供统一的跨层检索接口：

```python
coordinator.retrieve(
    query="用户喜欢的音乐",
    layers=["long", "episodic", "semantic"],
    top_k=3,
    threshold=0.3,
) -> dict  # {"long": [...], "episodic": [...], "semantic": [...]}
```

各层独立检索，结果按层分组返回。单层失败不影响其他层。

---

## 记忆系统的协同工作

一次完整对话回合中，四层记忆的协作流程：

```
用户输入 "帮我查一下明天天气"
        │
        ▼
1. 组装系统提示词
   build_system_prompt()
   → 读取 USER.md (用户自述)
   → 读取 MEMORY.md (长期记忆叙事)
   → 扫描 semantic_memory.json (语义记忆事实)
   → 拼接 AGENTS.md + SOUL.md + USER.md + MEMORY.md + 语义记忆段
        │
        ▼
2. episodic_retriever_node 检索情景记忆
   以用户输入为查询 → retrieve_from_episodic()
   → 相关历史 episode 注入 episodic_context
        │
        ▼
3. Agent 推理 → 执行工具 → 回答
   (可调用 search_memories / search_episodic / search_semantic 主动检索)
        │
        ▼
4. 回合结束
   → update_narrative(): 裸消息 + 旧 MEMORY.md → LLM → 新 MEMORY.md
   → commit_to_episodic(): checkpoint 摘要 → episodic_memory.json
```

这样，Agent 在本次对话中使用了过去的记忆（长期叙事 + 语义事实 + 情景回忆），同时本次对话的关键信息又被融合进各层记忆，供下次使用——形成一个自我改进的闭环。
