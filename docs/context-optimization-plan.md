# 上下文管理与记忆系统优化方案

> 创建日期：2026-06-30  
> 状态：进行中  
> 目标：降低 token 开销、提升缓存命中率、防止上下文溢出

---

## 背景与问题

当前架构采用"每轮全量注入 + 阈值截断 + 后台记忆写入"模式，存在以下问题：

1. **记忆叙事无限增长**：`get_narrative()` 每次请求全量注入所有记忆条目，无容量上限
2. **Prompt Cache 命中率低**：系统提示词中频繁变化的内容（记忆）穿插在稳定内容之间，导致缓存前缀断裂
3. **上下文截断偏保守**：75% 阈值 + 固定 5 轮保留，工具密集场景下可能溢出
4. **MemoryItem.history 膨胀**：YAML 文件中历史记录无限累积
5. **记忆消费者冗余处理**：每轮发送全量会话历史，而非增量消息

---

## 优化任务

### P0：系统提示词重组 — Prompt Cache 优化

**优先级**：最高（立即收益，改动最小）  
**目标**：最大化 Anthropic/OpenAI prompt caching 命中率，降低 90% 重复 input token 费用  
**状态**：已完成

**方案**：  
将系统提示词按"变化频率"分层排列，最稳定的放最前面，最频繁变化的放最后面。

```
当前顺序（缓存不友好）：
  行为规则(稳定) → 性格设定(稳定) → 用户自述(稳定) → 长期记忆(频繁变化) → skills(稳定) → macros(稳定)

优化后顺序（缓存友好）：
  行为规则(稳定) → 性格设定(稳定) → 用户自述(稳定) → skills(稳定) → macros(稳定) → 长期记忆(频繁变化)
```

**改动范围**：  
- `agent/prompts.py` 的 `_rebuild()` 函数
- 调整 `_cached_prompt` 和 `_cached_parts` 的拼接顺序
- 将 `get_narrative()` 移到最后

**验收标准**：
- [x] 系统提示词中 `get_narrative()` 内容排在 skills/macros 之后
- [x] `_cached_parts` 列表顺序与拼接顺序一致
- [x] 功能测试：系统提示词内容完整，无遗漏

---

### P1：记忆叙事容量上限 + 相关性排序

**优先级**：高（防止长期使用时 token 爆炸）  
**目标**：控制记忆叙事在系统提示词中的 token 占比  
**状态**：已完成

**方案**：  
1. `_format_narrative()` 加入容量上限（最多 40 条，约 3000 tokens）
2. 按 `latest_update_time` 降序排列，最近更新的记忆排在前面
3. 超出上限时附加提示："（还有 N 条较早记忆已折叠）"

**改动范围**：  
- `memory/narrative.py` 的 `_format_narrative()`
- `memory/memory_manager.py` 的 `show()` 方法（增加排序和限制参数）

**验收标准**：
- [x] 记忆条目超过 40 条时，系统提示词中只包含最近 40 条
- [x] 记忆按更新时间降序排列
- [x] 折叠提示正确显示剩余条目数

---

### P2：MemoryItem.history 裁剪

**优先级**：中（防止 YAML 膨胀）  
**目标**：限制历史记录条数，防止 YAML 文件无限增长  
**状态**：已完成

**方案**：  
在 `MemoryItem.update()` 中保留最近 5 条历史记录，超出时丢弃最早的。

**改动范围**：  
- `memory/memory_manager.py` 的 `MemoryItem.update()` 方法

**验收标准**：
- [x] 单条记忆的 history 列表不超过 5 条
- [x] 更新超过 5 次后，最早的记录被自动丢弃
- [x] 现有记忆数据不受影响（向后兼容）

---

### P3：上下文截断策略优化

**优先级**：中（防止上下文溢出）  
**目标**：更积极地管理上下文窗口，保留关键信息  
**状态**：已完成

**方案**：  
1. `TRIM_THRESHOLD` 从 0.75 降低到 0.6
2. `MIN_RECENT_TURNS` 改为动态计算（根据 ToolMessage 比例调整）
3. 引入"增量摘要"：截断后保留摘要 SystemMessage，下次截断时合并旧摘要 + 新旧消息

**改动范围**：  
- `agent/context_manager.py` 的常量和核心函数
- `chat.py` 中调用 `maybe_trim_checkpoint()` 的逻辑（可选）

**验收标准**：
- [x] 上下文占比超过 60% 时触发截断
- [x] 工具密集场景（平均每轮 3+ 工具调用）保留 3 轮，纯文本场景保留 6 轮
- [x] 增量摘要：新摘要包含旧摘要内容
- [x] 功能测试：长对话不丢失关键上下文

---

### P4：记忆消费者增量处理

**优先级**：中（降低记忆处理开销）  
**目标**：避免每轮发送全量会话历史给记忆消费者  
**状态**：已完成

**方案**：  
`chat.py` 中改为只发送本轮新增的消息（user message + final answer），而非全量 checkpoint 消息。

**改动范围**：  
- `api/routes/chat.py` 的 `_run_agent_turn()` 后处理部分
- 将 `messages_for_memory` 改为只包含本轮消息

**验收标准**：
- [x] 记忆消费者只接收本轮新增消息
- [x] 记忆 CRUD 决策正确（不遗漏重要信息）
- [x] 日志中 `send_history` 的消息数量等于本轮消息数（非全量）

---

### P5：上下文压缩质量提升（借鉴 Claude Code + Codex）

**优先级**：高（直接提升截断后的对话质量）  
**目标**：从"通用摘要"升级为"结构化交接文档"，保留具体实体，支持按需重读  
**状态**：已完成

**方案**：  
1. **结构化摘要模板**（借鉴 Claude Code compact）：用固定格式提取任务状态、关键实体、决策、错误，替代通用 200 字摘要
2. **自适应摘要长度**：根据消息数量动态调整每条消息的截取长度（100-400 字），简单对话短、复杂对话长
3. **实体提取与重读清单**（借鉴 Codex）：从被截断的消息中提取文件路径、URL，附在摘要末尾，模型在后续对话中可以用工具重新读取
4. **回退方案也包含实体提取**：`_summarize_old_messages()` 无 LLM 时同样追加实体清单

**改动范围**：  
- `agent/context_manager.py`：新增 `_extract_entities()`、`_COMPACT_PROMPT`，重写 `_llm_summarize()` 和 `_summarize_old_messages()`

**验收标准**：
- [x] LLM 摘要使用结构化模板（任务状态/关键实体/决策/错误）
- [x] 摘要长度根据对话复杂度自适应（无硬编码字数限制）
- [x] 被截断消息中的文件路径和 URL 被提取并附在摘要末尾
- [x] 无 LLM 回退方案同样包含实体提取结果

---

### P6：选择性记忆注入（RAG-like）— 长期演进

**优先级**：低（长期方向，需评估必要性）  
**目标**：从"全量注入"演进到"按需注入"  
**状态**：待规划

**方案**：  
1. 为每条记忆生成 embedding 向量，存入本地向量存储（如 ChromaDB）
2. 每轮对话时，用用户消息做语义检索，只取 top-10 相关记忆注入系统提示词
3. 即使积累 500 条记忆，每次请求也只注入最相关的 10 条（~750 tokens）

**改动范围**：  
- 新增 `memory/embeddings.py`（向量生成与存储）
- 新增 `memory/retrieval.py`（语义检索接口）
- `agent/prompts.py` 中 `get_narrative()` 替换为 `get_relevant_memories(query)`
- 依赖：chromadb 或 faiss-cpu

**验收标准**：
- [ ] 记忆检索响应时间 < 100ms
- [ ] 检索结果与用户消息语义相关
- [ ] 系统提示词中记忆部分 token 数稳定在 ~750 tokens 以内

---

## 实施顺序

```
P0 (Prompt Cache)  ──→  P1 (记忆容量上限)  ──→  P2 (history 裁剪)
                         ↓
                   P3 (截断策略)  ──→  P4 (增量处理)
                         ↓
                   P5 (RAG) — 评估后决定是否实施
```

P0-P2 可并行实施，改动独立且轻量。P3-P4 涉及核心流程，需逐步验证。P5 为长期方向。

---

## 预期收益

| 优化项 | 预期 token 节省 | 实施难度 | 立即收益 |
|--------|----------------|----------|----------|
| P0 Prompt Cache | 60-80% input tokens（缓存命中时） | 低 | 是 |
| P1 记忆容量上限 | 防止记忆部分超过 3000 tokens | 低 | 是 |
| P2 history 裁剪 | YAML 文件大小可控 | 极低 | 否（间接） |
| P3 截断策略 | 防止上下文溢出，减少 20-30% 冗余 | 中 | 是 |
| P4 增量处理 | 记忆处理 token 减少 80%+ | 低 | 是 |
| P5 RAG | 记忆部分稳定在 ~750 tokens | 高 | 长期 |
